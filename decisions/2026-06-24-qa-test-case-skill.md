# QAテストケース作成スキル（ck:qa）

## Context

AI にテストケースを出させると正常系に偏り、DB整合・回帰・仕様突合などの観点が頼まないと抜ける。
加えて、新機能（課題＋ソースが正解の基準）と移行（既存仕様＋既存挙動が基準）で「正解の基準」が違うのに同じ作り方をしていた。
台帳が仕様変更に追従せず「どのケースが今も生きているか」が分からなくなり、回帰テストの「この機能を触ったらあれを確認」が属人化していた。

共通スタンスは「動いているはず、を信用せず、意地悪なくらい疑ってかかる」。P1〜P7 のペルソナが各最低1観点を残すことで正常系バイアスを潰す。

既存スキルとの棲み分け: `ck:test` はユニットテスト「コード」、`ck:playwright-mcp-e2e` は E2E「コード」を生む。本スキルは「何をテストすべきか」=テストケース（観点・台帳）そのものを生む独立スキル。

## Decision

- **成果物**: `tests/qa/<機能名>.md` の機能単位 Markdown 台帳（1機能=1ファイル）。git 管理し、仕様変更を diff で検知する。CSV は必要時のエクスポート補助に留める。
- **二軸モード**: 起動時に「新機能 / 移行」を明示選択する（自動検出しない）。期待結果のオラクル（正解の基準）がモードで異なる:
  - 新機能 = 「あるべき仕様」。一次情報は課題/チケット＋実装ソース（課題が正、ソースは実装意図の確認用）。重点ペルソナ P1〜P4。
  - 移行 = 「旧挙動との一致」。一次情報は旧仕様書＋旧システムの実挙動＋旧データ。実装＝正しい仕様とは限らない前提。重点ペルソナ P5〜P7。
- **移行モードの旧バグ扱い**: オラクルは「旧と一致」だが、旧挙動が明らかにバグの場合は一致させず `要確認` フラグを立て人間判断に回す（P7仕様懐疑者の役割）。
- **ペルソナ固定リスト（P1〜P7）**: スキル本文に固定。各ペルソナは最低1観点を残す。観点が無いペルソナは削除せず「該当なし＋理由」を明記し、抜けを可視化する。
- **P4の裏側確認必須**: 期待結果は「保存→画面表示」で終わらせず「裏のテーブルに値が入ったか（例: orders に1件 INSERT）」まで書かせる。
- **サマリ2階層**:
  - ① 各台帳ファイル先頭ヘッダ（モード/一次情報/ケース総数/active・obsolete内訳/ペルソナ別カバレッジ/要確認件数/最終更新）。
  - ② `tests/qa/README.md` の横断 index（全機能の台帳リンク・active・obsolete・要確認の集計・更新日）。「今生きているケース」の総覧。
  - ペルソナ別カバレッジ（P1〜P7 各何件）をサマリに含め、正常系バイアスを数字で自己点検できるようにする。
- **陳腐化対策（鮮度管理）**: 既存台帳がある機能で再実行すると照合パスが走る。現在の一次情報を読み直し既存ケースと差分を取り、合わなくなったケースを `要確認` / `obsolete` に降格する。勝手に消さず人間判断に回す。状態は `active` / `要確認` / `obsolete` の3値。
- **回帰の属人化解消**:
  - 各ケースに関連モジュール/ファイルをタグ付けし、変更ファイルから関連ケースを逆引きできるようにする。
  - 各台帳に「回帰トリガー」節（P6担当）を設け「この機能を触ったら確認すべき周辺ケース」を明文化する。

### 改善版（2026-06-24 v2 discuss）— 72点→85点クラスへの引き上げ

- **技法軸（HOW）の追加**: ペルソナ（WHO）だけでは体系的網羅を担保できないため、テスト技法軸を直交で追加。Step 4.5「技法チェック」を新設し、台帳に**技法タグ列**を持たせる。採用5技法に固定: 境界値分析・同値分割・状態遷移・デシジョンテーブル・組合せ/ペアワイズ。
- **検証レンズへの再構成**: 「ペルソナ」の軸が不揃い（P1/P2/P5＝利用者像、P3/P4/P6/P7＝観点の擬人化）だったため、全7つを **「検証レンズ（疑う視点）」L1〜L7** に1軸統一して再定義。テーブル構造・件数は維持し既存台帳と互換。
  - L1 直感操作で壊す / L2 高速大量入力で壊す / L3 境界・不正・権限で壊す / L4 裏のDB整合を疑う / L5 既存データ投入で壊す / L6 周辺の回帰で壊す / L7 仕様との不一致を暴く（旧 P1〜P7 に対応）。
- **優先度列（リスクベース）**: テーブルに優先度列を追加。High＝データ破壊・金額・権限／Mid＝主要導線／Low＝表示・軽微。
- **ケースID横断一意化**: `<機能略号>-連番`（例 `ORD-C1`、`STK-C3`）。略号は台帳ヘッダで宣言。回帰トリガーの他台帳参照が衝突しないようにする。
- **完了条件に質ゲート**: 形式充足（全レンズが観点を持つ）だけで完了とせず、(1) **技法別カバレッジ**もサマリに出し「境界値0件」等の穴があれば未完了、(2) **水増し禁止**を明文化（観点が無いレンズは堂々と「該当なし＋理由」にせよ。水増しの1件は該当なしより悪い、と価値観を反転）。
- **改修後のテーブル列**: `| ID | レンズ | 技法 | 優先度 | 観点 | 前提/手順 | 期待結果（裏側確認含む） | 関連ファイル | 状態 |`

### 品質強化（2026-06-24 v3 discuss）— 非機能観点の追加

- **検証レンズを L1〜L8 に拡張**: 非機能観点（並行性・トランザクション整合・冪等性・性能）を**観点起点グループ**のレンズとして追加（独立アクティビティ・技法軸ではなくレンズ＝最強の enforcement に乗せる）。性質で2本に分割し、あわせて番号をグループ連番に全面振り直し（運用台帳ゼロの今が最後の好機）:
  - アクター起点（誰が壊すか）: **L1** 直感操作 / **L2** 高速大量入力 / **L3** 権限・不正・悪意 / **L4** 既存データ投入（移行担当）
  - 観点起点（どこを疑うか）: **L5** 裏のデータ整合 / **L6** 並行・負荷で壊す / **L7** 整合・障害復旧で壊す / **L8** 周辺の回帰
  - 旧 L1〜L6 からの対応: 旧L1=新L1, 旧L2=新L2, 旧L3=新L3, 旧L5既存データ=新L4, 旧L4裏DB整合=新L5, 旧L6回帰=新L8。新設は L6/L7。
- **WHO/HOW 直交の優先ルール明記**: レンズ（WHO）が主分類、技法（HOW）は補助タグ。同義になってもケースは1件。レンズ別／技法別カバレッジは合計がケース総数を超えない。
- **技法に「エラー推測」を追加（6種化）**: ただし経験則技法のため**カバレッジ穴判定の対象外**（境界値・状態遷移のような0件＝穴判定はしない）。
- 仕様突合は引き続き Step 4.7 の独立アクティビティ（レンズ化しない）。重点配分: 新機能=L1〜L5を厚く／移行=L4〜L8を厚く（full は一巡）。

## Consequences

- 新スキル `ck:qa`（仮）を二層構造で追加: 本体 `packages/ck/src/skills/qa/body.md` ＋ スタブ `plugins/ck/skills/qa/SKILL.md`。`ck skill doctor` で整合性確認、新スタブ認識にはセッション再起動が必要。
- 成果物ディレクトリ `tests/qa/` が新設される。`tests/qa/README.md` が横断 index として機能。
- 台帳には実在の顧客名・案件名・認証情報を書かない運用ルール（CLAUDE.md）が適用される。
- `/plan` で body.md の章構成（モード分岐・ペルソナ走査・照合パス・サマリ生成・回帰トリガー）とスタブのフロントマター（allowed-tools 等）を詰める。

### 2026-06-24 v3 discuss（非機能観点の追加・品質強化）

スキル全体の点数評価（忖度なしレビュー）で qa の品質上の穴として「非機能観点の欠落・WHO/HOW 直交の一部破れ・エラー推測技法の不在・レンズ番号の不揃い」を指摘。ユーザー方針＝「qa の質は落としたくない。トークンが重いのは品質直結なので許容」を前提に、レンズ体系の根幹を触るため discuss で4論点を確定。前提事実: `tests/qa/` の運用台帳はリポジトリ内ゼロ（番号振り直しの後方互換代償が無い最後のタイミング）。

#### Decision: 非機能観点を「観点起点レンズ」として追加する（案A）
- **Constraint**: 並行性・トランザクション整合・冪等性・性能を、Step 4.7 型の独立アクティビティや技法軸ではなく、L4/L6 と同じ**観点起点グループのレンズ**として追加する。カバレッジ集計と「該当なし＋理由」の強制対象に含める。
- **Why**: レンズはカバレッジ集計と該当なし強制で enforcement が効く唯一の軸。品質直結ゆえ最も強い強制をかけたい。観点起点グループ（"どこを疑うか"）に入れる限り、v2 で仕様懐疑をレンズから外した「軸の不揃い」問題は再発しない（L4/L6 は既に観点起点）。
- **Rejected alternatives**: B＝Step 4.7 型の独立アクティビティ（ケース単位の網羅強制が緩い）／C＝既存 L3/L4 にチェックリスト内包（埋もれて運頼みに戻る）／D＝技法軸に追加（技法は"値の網羅"軸で、競合は値ではないため破れる）。

#### Decision: 非機能を2レンズに分割し、番号体系をグループ連番に振り直す（2本＋案イ）
- **Constraint**: 非機能を性質で2本に分ける ―「並行・負荷で壊す」（レース/同時更新の競合/排他・ロック/性能・件数限界）と「整合・障害復旧で壊す」（トランザクション部分失敗/ロールバック/冪等性/リトライ/タイムアウト後の状態）。あわせてレンズ番号を**グループと連番が一致するよう全面振り直し**する:
  - アクター起点（誰が壊すか）: L1 直感操作 / L2 高速大量入力 / L3 権限・不正・悪意 / L4 既存データ投入（移行担当）
  - 観点起点（どこを疑うか）: L5 裏のデータ整合 / L6 並行・負荷 / L7 整合・障害復旧 / L8 周辺の回帰
- **Why**: 薄い1本より性質の違う2本の方が網羅を細かく強制でき品質優先と一致。番号振り直しは番号とグループの不一致による運用事故を消す。**運用台帳ゼロの今しかできず（v2 で見送った理由＝既存資産の代償が今は存在しない）**、これが最後の好機。
- **Rejected alternatives**: 1レンズに非機能を全部詰める（広すぎて雑に畳まれる）／案ロ＝既存 L1〜L6 を触らず非機能を L7/L8 末尾追記（不揃いが恒久化）。

#### Decision: WHO/HOW が同義になる場合の優先ルールを明記
- **Constraint**: レンズ（WHO）をケースの主分類、技法（HOW）は同一ケースに付す補助タグとする。両者が同義（例: L3 権限 × デシジョンの権限マトリクス網羅）でも**ケースは1件**。レンズ別／技法別カバレッジは同一ケースを2軸から見た数字であり、合計はケース総数を超えない。
- **Why**: 現行 body は「二重計上しない」とのみ書き優先順位が無く、運用者が「どちらで数えるか」迷う。主分類＝レンズと固定すれば迷いと二重計上が同時に消える。

#### Decision: 技法に「エラー推測」を6つ目として追加（穴判定からは除外）
- **Constraint**: 技法タグに「エラー推測」を追加し6種にする。ただし他5技法（体系的網羅）と異なり経験則ベースのため、**カバレッジ穴判定の対象外**とする（「エラー推測0件＝穴」とは判定しない）。
