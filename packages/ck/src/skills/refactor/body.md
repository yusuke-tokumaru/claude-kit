# refactor

**挙動を変えずに**コードの構造を改善する。implement / debug が「ついでにリファクタ禁止」で追い出した構造改善の受け皿。

**位置づけ**: 単独で起動する（機能追加・バグ修正のついでに行わない）。完了後はキット標準の連鎖に合わせて `/review`（自己レビュー）→ `/commit`（またはユーザーのコミット）の順に進む。

**コア原則: 挙動不変。テスト緑を保ったまま小さな変形を1つずつ積む。1変形 = 1検証。**

## 前提

- **新機能・挙動変更・バグ修正を混ぜない**。作業中にバグを見つけたら `ck note` で症状・再現箇所を記録して `/debug` に回す（このスキル内で直さない）
- アーキテクチャレベルの再設計（モジュール境界・データモデルの変更）は設計判断なので `/plan`（必要なら `/discuss`）の責務。ここでは扱わない

**非対話実行（ユーザーに質問できない自動実行の場合）**: セーフティネットの選択（Step 2）は推奨案（特性テスト先行）を採用する。テストが書けない状況なら型チェック＋実挙動確認で小規模に進め、置いた仮定を最終報告で列挙する。3回失敗ルールは緩和しない。

## Step 1: 対象とゴールを確定する

- `$ARGUMENTS` から対象（ファイル・モジュール）と狙い（重複除去・命名改善・関数抽出・依存整理など）を特定する。不明なら1行で確認する（非対話時は会話文脈・直近の変更箇所から最も蓋然性の高い対象を推定して進め、推定である旨を最終報告に明記する）
- ゴールを「何がどう変われば完了か」の1〜2文にする（例: 「3ファイルに重複する集計ロジックを1つの純関数に集約」）
- `git status --short` で作業ツリーがクリーンか確認する。未コミット変更が混在すると挙動不変の検証が濁るため、残っていれば先に `/commit` を案内する

## Step 2: セーフティネットを確認する

対象コードを覆う既存テストを実行し、**緑を確認してから**着手する。

テストが無い・対象を覆っていない場合は `AskUserQuestion` で確認する:

- 「特性テストを先に作る（推奨）」— 現在の挙動をそのまま固定するテストを最小限作成してから着手する。期待値は「あるべき仕様」ではなく**現状の出力**にする（挙動不変の検証が目的のため）
- 「型チェック＋実挙動確認で進める」— 小規模な変形に限る

## Step 3: 小さな変形を1つずつ適用する

rename → extract → move のように小さい変形から積む。各変形で以下を厳守する:

1. **1種類の変形だけ**適用する（rename しながら extract しない）
2. テスト・型チェックを実行し**緑を確認**する
3. 緑になってから次の変形へ進む。赤のまま変形を重ねない

**巻き戻しの手段**: コミットは自動で作らない方針のため、巻き戻しは「直前の1変形」を Edit で逆適用して行う（1変形=1検証を守っていれば逆適用は常に機械的にできる）。それを超える巻き戻し（複数変形前の状態に戻したい）が必要になった時点で作業を止め、状況を報告してユーザーに委ねる（`git stash`・途中コミット等の作業ツリー操作はユーザー承認なしに行わない）。

リネーム・移動は `Grep` で全使用箇所を洗い出してから変える（1箇所だけ変えて他が旧名のまま、が典型事故。同種パターンは全箇所に横展開する）。

## Red Flags — STOP

これらの思考が浮かんだら止まる:

- 「ついでにこのバグも直す」→ 挙動不変違反。`ck note` に記録して `/debug` へ
- 「テストの期待値を変えれば通る」→ それは挙動が変わった証拠。変形を巻き戻す
- 「一気に書き直した方が早い」→ 小刻み違反。戻れない地点を作らない
- 「公開インターフェース（エクスポート・API・CLI 引数）も変えたい」→ 呼び出し側への影響は設計判断。`/plan` へ
- **同一変形の試行が3回失敗** → 4回目を試さない。対象の設計自体を疑い、ユーザーと議論する

## 制約

- 挙動・公開インターフェースを変えない（テストの期待値変更＝違反のシグナル）
- 既存テストを削除・弱体化しない（特性テストの追加は可）
- `git add` を自動実行しない（コミットは `/commit` またはユーザーに委ねる）
- 変形と無関係な整形（フォーマッタの一括適用など）をバンドルしない

## 完了条件

ゴールに掲げた構造改善が完了し、全テスト・型チェックが緑の状態。変更サマリ（何をどこへ動かしたか・変えていないこと）を報告し、`/review` → `/commit` を案内する。
