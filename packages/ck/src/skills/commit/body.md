# commit

未コミットの作業ツリー差分を**論理単位に分割してコミットする**。implement（既定=コミットしない）で積み上がった変更を、レビュー可能な粒度の履歴に変換する。

**位置づけ**: `/implement`（または手作業）で変更が完了したあと、`/pr` の前段。`/pr` は未コミット変更があると停止するため、その間を埋める。

**コア原則: 1コミット = 1つの理由。「同じ理由で変わったファイル」だけを同じコミットに入れる。**

## 非対話実行時

分割案の確認（Step 3）は推奨案を採用して進めてよい（コミット作成はこのスキルの本務であり、起動自体を承認とみなす）。ただし採用した分割と除外したファイルを最終報告で列挙する。push は非対話でも実行しない。

- デフォルトブランチ上にいる場合（Step 1）: 非対話ではブランチを切らず、コミットも作成せずに停止して報告する（ブランチ戦略はユーザー判断）
- 1ファイルに複数の論理変更が混在している場合（Step 2）: 非対話では hunk 分割を試みず、そのファイルは「混在のため保留」として除外し、残りのグループのみコミットして報告する

## Step 0: 引数を解釈する

`$ARGUMENTS` に分割方針や対象スコープ（例: 「スキル本体だけ」「2コミットに分けて」）が指定されている場合は、Step 2 の分割案に反映する。指定がなければ全差分を対象とする。

## Step 1: 差分を把握する

```bash
git status --short          # 変更・未追跡ファイル
git diff --stat             # 変更規模
git log -5 --oneline        # 既存のメッセージ規約を確認
```

- 未コミット変更が無ければ「コミット対象がありません」と伝えて終了する
- CLAUDE.md にコミット規約（言語・プレフィックス等）があれば従う。無ければ `git log` の既存スタイルに合わせる
- デフォルトブランチ上にいる場合はその旨を伝え、ブランチを切るか確認してから進む。デフォルトブランチは `git symbolic-ref refs/remotes/origin/HEAD` で判定する（出力の `refs/remotes/origin/` 以降がブランチ名。エラー時は `git branch -r` の一覧から特定し、リモートが無いローカルリポジトリでは main / master の存在で判断する）

## Step 2: 論理単位に分割する

分割の一次情報（あれば）:

- `plans/<date>-<slug>.md` — プランのステップ構成をそのままコミット分割に写す（1ステップ = 1コミット）
- `decisions/<date>-<slug>.md` — 変更理由の言語化に使う

無ければ差分の内容を読み、「同じ理由で変わったか」でファイルをグルーピングする。

分割案を以下の形式で提示する:

```
1. <コミットメッセージ（命令形）>
   - path/to/a.ts
   - path/to/a.test.ts
2. <コミットメッセージ>
   - path/to/b.md
除外（コミットしない）: .env.local（機密）・tmp/（一時ファイル）
```

- **1ファイルに複数の論理変更が混在**している場合はそのまま進めず、混在の内容を提示してユーザーに分け方を相談する（分け方の合意後、Step 4 の hunk 分割手順で部分ステージする）
- 機密らしきファイル（`.env*`・鍵・トークンを含むファイル）と一時ファイルは分割案から除外し、除外理由を明示する

## Step 3: 分割案を確認する

`AskUserQuestion` で分割案全体を確認する（グループごとに個別確認はしない）:

- 「この分割でコミットする」
- 「分割・メッセージを調整する」

調整の指示を受けたら反映して再提示する（合意できるまで繰り返してよい。確認の単位は常に分割案全体）。

## Step 4: コミットを実行する

グループごとに順番に実行する:

```bash
git add <グループのファイルのみ>   # -A / . / -u は使わない
git commit -m "<命令形メッセージ>"
```

- pre-commit フック等で失敗した場合は出力を提示する（`--no-verify` で回避しない）。フックがファイルを自動修正した場合はその差分を同じグループに含めて再実行する。原因の修正に**コード編集が必要な場合**は本スキルでは編集せず、失敗内容と修正方針を報告してユーザー（または `/implement`・`/debug`）に委ね、修正後に再実行する

**1ファイル内の hunk 分割が必要な場合**（Step 2 で合意した分け方に従う）: `git add -p` は対話 TUI のため使わない。patch 経由で部分ステージする:

```bash
# リダイレクトではなく --output を使う（出力を加工するシェルフック環境でも patch が壊れない）
git diff --output=<作業ツリー外の一時パス>/split.patch <ファイル>
# patch を Read で確認してから Edit で編集し、先にコミットするグループの hunk（@@ ブロック単位）だけを残す。
# hunk 内の行は改変しない（壊れた patch は apply が拒否する）
git apply --cached <作業ツリー外の一時パス>/split.patch
git diff --cached <ファイル>   # ステージ内容が意図した hunk のみか確認してからコミットする
```

一時 patch はスクラッチパッド等の作業ツリー外に置き、コミット対象に含めない。残りの hunk は作業ツリーに残るので、次のグループで通常どおり `git add <ファイル>` する。

## Step 5: 検証と報告

```bash
git status --short              # 意図した除外だけが残っているか
git log --oneline -<作成数>     # 作成したコミットの確認
```

作成したコミット一覧と、残した未コミットファイル（除外分）を報告する。

## 制約

- `git add -A` / `git add .` / `git commit -a` を使わない（無関係ファイルの混入防止）
- push しない（`/pr` の責務。push はユーザー承認後のみ）
- `--amend`・rebase 等の履歴改変をしない（既存コミットには触れない）
- `--no-verify` でフックを回避しない
- 機密情報を含むファイルをステージしない。検知したら除外して報告する
- **コミットメッセージにも秘匿情報ガードを適用する**: メッセージは push でリモートに公開される。plans/・decisions/ 由来の文言から実在の顧客名・案件名・認証情報・社内 URL を転記せず、一般化した表現に置き換える（例: 「○○様向け」→「特定顧客向け」）

## 完了条件

分割案どおりのコミットが作成され、`git status` に意図した除外分のみが残っている状態。完了後は `/pr` を案内する。
