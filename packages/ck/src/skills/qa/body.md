# qa

「何をテストすべきか」=QAテストケース台帳を生成・更新する。テストコードは生まない（ユニットは `/test`、UI E2E は `/playwright-mcp-e2e` に委譲）。

**位置づけ**: `/test`・`/playwright-mcp-e2e` の前段または並走。台帳でケースを設計してからコード化する流れを想定する。

**共通スタンス**: 「動いているはず」を信用しない。意地悪なくらい疑ってかかる。P1〜P7 の各ペルソナが最低1観点を残し、観点が無いペルソナも黙ってスキップせず「該当なし＋理由」を明記する。これにより正常系バイアスを構造的に潰す。

成果物は機能単位の Markdown 台帳 `tests/qa/<機能名>.md`（kebab-case）と、横断 index `tests/qa/README.md`。

## 非対話実行時の振る舞い

サブエージェント等から呼ばれ `AskUserQuestion` が使えない場合は止まらない。モードは一次情報から推定する（課題/チケット主体なら新機能、旧システム/旧データ参照主体なら移行）。推定したモードと、本来ユーザーに確認すべき判断（オラクルの選択など）は仮定として台帳ヘッダに明記し、不確実なケースは `要確認` に記録して人間判断へ回す。

## Step 1: モード選択

`AskUserQuestion` で必ず明示選択させる（自動検出しない）。モードで「期待結果の正解の基準（オラクル）」が根本的に変わる。

- **新機能**: これから作る/作った機能のテスト。オラクルは「あるべき仕様」。
- **移行**: 旧システムから移す機能のテスト。オラクルは「旧システムの挙動との一致」。

## Step 2: 一次情報の収集（モード別オラクル）

選んだモードに応じて一次情報を集める。`Task`（Explore サブエージェント）・`Read`・`Grep`・`Glob` を使う。

| モード | 集める一次情報 | 期待結果の根拠（オラクル） |
|---|---|---|
| 新機能 | 課題/チケット ＋ 実装ソース ＋ DBスキーマ/マイグレーション | 「あるべき仕様」。課題が正、ソースは実装意図の確認用 |
| 移行 | 旧仕様書 ＋ 旧システムの実挙動 ＋ 旧データのサンプル | 「旧挙動との一致」。**実装＝正しい仕様とは限らない**前提 |

一次情報が不足している場合はユーザーに確認してから進む。期待結果を「あるべき姿」と「実装の現状」のどちらに合わせるかが曖昧なときも確認する。

## Step 3: 既存台帳の照合（鮮度管理）

`tests/qa/<機能名>.md` が既に存在する場合は読み込み、Step 2 で集めた現在の一次情報と差分照合する。

- 現仕様と矛盾するケース → 状態を `要確認` に降格し、何が変わったかをメモする。
- 機能ごと廃止されたケース → `obsolete` に降格する。
- **既存ケースを勝手に削除しない。** 降格に留め、最終判断は人間に残す。

存在しない場合は新規作成に進む。照合では `git log`/`git diff`（Bash）で前回更新日以降に関連ファイルが変更されていないかを洗い出し、変更があったケースを優先的に再判定する。

## Step 4: ペルソナ走査（P1〜P7）

各ペルソナで観点を洗い出す。**全ペルソナで最低1観点**を出す。観点が無いペルソナは削除せず「該当なし＋理由」を明記する（抜けの可視化）。

- モードで重点配分する：新機能は P1〜P4 を厚く、移行は P5〜P7 を厚く。ただし全ペルソナを必ず一巡する。
- **P4 は期待結果に裏側確認を必須化**：「保存した→画面に出た」で終わらせず、「裏のテーブルに値が入ったか（例: `orders` に1件 INSERT されたか）」まで書く。
- 移行モードで旧挙動が明らかにバグの場合：旧と一致させず、そのケースを `要確認` にして人間判断へ回す（P7 の役割）。

## Step 5: 台帳の生成・更新

`tests/qa/<機能名>.md` を `Write`/`Edit` で生成・更新する。下記「台帳フォーマット」に従う。

- 各ケースに **関連モジュール/ファイル** をタグ付けする（変更ファイルからケースを逆引きできるように）。タグ付けの裏取りには `grep`/`git`（Bash）で実際の参照箇所を確認する。
- **回帰トリガー節**（P6 担当）に「この機能の◯◯を変更したら → 確認すべきケースID / 他台帳のケース」を明文化する。頭の中の暗黙知をファイルに固定する。

## Step 6: サマリ更新

2階層のサマリを更新する。

1. **台帳ファイル先頭ヘッダ**：モード・一次情報・ケース総数（状態別内訳）・**ペルソナ別カバレッジ（P1〜P7 各何件）**・要確認件数・最終更新日。
2. **横断 index `tests/qa/README.md`**：全機能の台帳リンク・active/要確認/obsolete の集計・更新日。無ければ新規作成、あれば該当行を更新する。

ペルソナ別カバレッジは「正常系に寄っていないか」を数値で自己点検するためのもの。そのモードの重点ペルソナ（新機能=P1〜P4／移行=P5〜P7）が極端に少なければ観点を見直す。重点外ペルソナが該当なしになるのは正常（例: 新機能モードの P5）。

## 台帳フォーマット

```markdown
# <機能名> QAテストケース台帳
- モード: 新機能 | 移行
- 一次情報: <課題#123・src/order/* / 旧仕様書・旧システム挙動・旧データ>
- ケース総数: N（active A / 要確認 W / obsolete O）
- ペルソナ別: P1×n P2×n P3×n P4×n P5×n P6×n P7×n
- 要確認: W件 / 最終更新: YYYY-MM-DD

## テストケース
| ID | ペルソナ | 観点 | 前提/手順 | 期待結果（裏側確認含む） | 関連ファイル | 状態 |
|----|---------|------|----------|------------------------|------------|------|
| C1 | P3 | 数量0送信 | 数量欄に0で送信 | バリデーションで拒否、orders 未作成 | src/order/validate.ts | active |
| C2 | P4 | 保存後のDB反映 | 正常値で保存 | orders に1件 INSERT、金額が一致 | src/order/save.ts | active |

## 回帰トリガー（P6）
- 在庫引当ロジックを変更したら → C2, C5 と stock.md の C3 を再確認
- 受注フォームに項目追加したら → C1, C7（バリデーション）を再確認

## 該当なしペルソナと理由
- P5: 本機能は移行対象外のため該当なし
```

横断 index `tests/qa/README.md`:

```markdown
# QAテストケース台帳 index
| 機能 | 台帳 | active | 要確認 | obsolete | 更新日 |
|------|------|--------|--------|----------|--------|
| 受注入力 | [order-entry.md](order-entry.md) | 10 | 1 | 2 | 2026-06-24 |
```

## ペルソナ定義（P1〜P7）

各ペルソナの「疑う点」。最低1観点を残す。

| # | ペルソナ | 疑う点（代表観点） |
|---|---------|------------------|
| P1 | 新人ユーザー | 説明を読まず直感で操作。誤クリック・空送信・連打で壊れないか |
| P2 | ベテラン現場担当 | キーボードで高速・大量入力。Tab遷移／ショートカット／IME変換中のEnter |
| P3 | 悪意ある操作者 | 境界値・不正値・権限外・二重送信。バリデーションや排他制御が効くか |
| P4 | データ整合性監査役 | 画面を信用せず裏のDBテーブルを直接確認。CRUDの整合性 |
| P5 | 移行担当者 | 旧システムの既存データを投入。欠損・異形式・文字コード・件数の一致 |
| P6 | 回帰デグレ番人 | 「今まで動いていた何か」が壊れていないか。周辺機能やF5後の挙動 |
| P7 | 仕様懐疑者 | 「実装＝正しい仕様」を信用しない。一次情報（課題・仕様書）と挙動を突合 |

## 状態定義

| 状態 | 意味 |
|---|---|
| `active` | 現仕様で有効なケース |
| `要確認` | 仕様変更で再判定が必要、または移行で旧挙動がバグ疑い。人間判断待ち |
| `obsolete` | 廃止されたケース。記録として残し削除しない |

## モード別オラクル早見表

- **新機能**: 期待結果は「あるべき仕様」に合わせる。実装の現状とズレたら実装側の不具合を疑う。
- **移行**: 期待結果は「旧システムの挙動」に合わせる。ただし旧挙動が明らかにバグなら一致させず `要確認` で人間へ。

## 制約

- 台帳に実在の顧客名・案件名・社内システムの認証情報を書かない（git 管理されるため）。
- 既存ケースを勝手に削除しない。`要確認`/`obsolete` への降格に留め、最終判断は人間に残す。
- テストコードは生成しない。コード化が必要なら `/test`・`/playwright-mcp-e2e` を案内する。
- ペルソナの観点漏れを黙殺しない。該当なしは必ず理由を明記する。
- 主成果物は Markdown 台帳。CSV が必要な場合は台帳から書き出す補助エクスポートに留め、CSV を一次成果物にしない。

## 完了条件

`tests/qa/<機能名>.md` と2階層サマリ（先頭ヘッダ＋`tests/qa/README.md` index）を更新し、P1〜P7 すべてが観点または「該当なし＋理由」を持つ状態。コード化が必要なら `/test`・`/playwright-mcp-e2e` を案内して完了。
