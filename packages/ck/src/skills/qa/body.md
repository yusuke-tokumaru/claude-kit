# qa

「何をテストすべきか」=QAテストケース台帳を生成・更新する。テストコードは生まない（ユニットは `/test`、UI E2E は `/playwright-mcp-e2e` に委譲）。

**位置づけ**: `/test`・`/playwright-mcp-e2e` の前段または並走。台帳でケースを設計してからコード化する流れを想定する。

**共通スタンス**: 「動いているはず」を信用しない。意地悪なくらい疑ってかかる。L1〜L7 の各検証レンズが最低1観点を残し、観点が無いレンズも黙ってスキップせず「該当なし＋理由」を明記する（WHO 軸）。さらに各ケースに**テスト技法**（境界値・同値分割など）をタグ付けして体系的網羅も担保する（HOW 軸）。この WHO×HOW の二軸で正常系バイアスを構造的に潰す。

成果物は機能単位の Markdown 台帳 `tests/qa/<機能名>.md`（kebab-case）と、横断 index `tests/qa/README.md`。

## 非対話実行時の振る舞い

サブエージェント等から呼ばれ `AskUserQuestion` が使えない場合は止まらない。モードは一次情報から推定する（課題/チケット主体なら新機能、旧システム/旧データ参照主体なら移行）。機能略号も機能名から推定する。推定したモード・略号と、本来ユーザーに確認すべき判断（オラクルの選択など）は仮定として台帳ヘッダに明記し、不確実なケースは `要確認` に記録して人間判断へ回す。

## Step 1: モード選択

`AskUserQuestion` で必ず明示選択させる（自動検出しない）。モードで「期待結果の正解の基準（オラクル）」が根本的に変わる。

- **新機能**: これから作る/作った機能のテスト。オラクルは「あるべき仕様」。
- **移行**: 旧システムから移す機能のテスト。オラクルは「旧システムの挙動との一致」。

あわせて**機能略号**（横断一意なケースID用、例: 受注入力=`ORD`、在庫=`STK`）を決める。kebab-case 機能名から導出し、対話可能なら確認、非対話なら推定して台帳ヘッダに明記する。

## Step 2: 一次情報の収集（モード別オラクル）

選んだモードに応じて一次情報を集める。`Task`（Explore サブエージェント）・`Read`・`Grep`・`Glob` を使う。

| モード | 集める一次情報 | 期待結果の根拠（オラクル） |
|---|---|---|
| 新機能 | 課題/チケット ＋ 実装ソース ＋ DBスキーマ/マイグレーション | 「あるべき仕様」。課題が正、ソースは実装意図の確認用 |
| 移行 | 旧仕様書 ＋ 旧システムの実挙動 ＋ 旧データのサンプル | 「旧挙動との一致」。**実装＝正しい仕様とは限らない**前提 |

一次情報が不足している場合はユーザーに確認してから進む。期待結果を「あるべき姿」と「実装の現状」のどちらに合わせるかが曖昧なときも確認する。

## Step 3: 既存台帳の照合（鮮度管理）

`tests/qa/<機能名>.md` が既に存在する場合は読み込み、Step 2 で集めた現在の一次情報と差分照合する。

- 現仕様と矛盾するケース → 状態を `要確認` に降格し、何が変わったかをメモする。
- 機能ごと廃止されたケース → `obsolete` に降格する。
- **既存ケースを勝手に削除しない。** 降格に留め、最終判断は人間に残す。

存在しない場合は新規作成に進む。照合では `git log`/`git diff`（Bash）で前回更新日以降に関連ファイルが変更されていないかを洗い出し、変更があったケースを優先的に再判定する。

## Step 4: 検証レンズ走査（L1〜L7）= WHO 軸

各レンズで観点を洗い出す。**全レンズで最低1観点**を出す。観点が無いレンズは削除せず「該当なし＋理由」を明記する（抜けの可視化）。

- モードで重点配分する：新機能は L1〜L4 を厚く、移行は L5〜L7 を厚く。ただし全レンズを必ず一巡する。
- **L4 は期待結果に裏側確認を必須化**：「保存した→画面に出た」で終わらせず、「裏のテーブルに値が入ったか（例: `orders` に1件 INSERT されたか）」まで書く。
- 移行モードで旧挙動が明らかにバグの場合：旧と一致させず、そのケースを `要確認` にして人間判断へ回す（L7 の役割）。
- **水増し禁止**：観点が本当に無いレンズは堂々と「該当なし＋理由」にする。無理にひねり出した薄い1件は、該当なしより悪い（網羅の演出になる）。

## Step 4.5: 技法チェック（HOW 軸）

レンズ走査で出した各ケースに、下記5技法のいずれか（複数可）をタグ付けする。レンズは「誰が壊しに来るか」、技法は「どう網羅するか」で直交する。レンズ走査だけだと境界値などが思いつき頼みになるため、技法の観点で取りこぼしを点検する。

| 技法 | 何を担保するか |
|---|---|
| 境界値分析 | 最小・最大・直前直後（0, 1, 上限, 上限+1, 空, NULL）で壊れないか |
| 同値分割 | 入力を有効/無効クラスに分け各クラス代表を1件ずつ |
| 状態遷移 | 下書き→確定→取消など状態の正常/異常な遷移、二重遷移 |
| デシジョンテーブル | 条件の組合せ（権限×ステータス×フラグ）の網羅 |
| 組合せ・ペアワイズ | 多パラメータ時の代表的な組合せ（全網羅が無理な場合） |

- 数値・日付・文字列長・件数を扱う機能で**境界値が0件なら穴**とみなし、観点を追加する。
- 状態を持つ機能（ワークフロー・申請承認等）で状態遷移が0件なら同様に見直す。

## Step 5: 台帳の生成・更新

`tests/qa/<機能名>.md` を `Write`/`Edit` で生成・更新する。下記「台帳フォーマット」に従う。

- 各ケースの **ID** は `<機能略号>-C<連番>`（例 `ORD-C1`）。機能内連番だが略号で横断一意になり、回帰トリガーの他台帳参照が衝突しない。
- 各ケースに **技法**・**優先度**（High/Mid/Low）・**関連モジュール/ファイル** をタグ付けする。
  - 優先度: **High**＝データ破壊・金額・権限に関わる ／ **Mid**＝主要導線 ／ **Low**＝表示・軽微。
  - 関連ファイルは変更ファイルからケースを逆引きするため。タグ付けの裏取りには `grep`/`git`（Bash）で実際の参照箇所を確認する。
- **回帰トリガー節**（L6 担当）に「この機能の◯◯を変更したら → 確認すべきケースID / 他台帳のケース」を明文化する。頭の中の暗黙知をファイルに固定する。

## Step 6: サマリ更新

2階層のサマリを更新する。

1. **台帳ファイル先頭ヘッダ**：モード・機能略号・一次情報・ケース総数（状態別内訳）・**レンズ別カバレッジ（L1〜L7 各何件）**・**技法別カバレッジ（各技法何件）**・要確認件数・最終更新日。
2. **横断 index `tests/qa/README.md`**：全機能の台帳リンク・active/要確認/obsolete の集計・更新日。無ければ新規作成、あれば該当行を更新する。

レンズ別・技法別カバレッジは「正常系に寄っていないか」を数値で自己点検するためのもの。そのモードの重点レンズ（新機能=L1〜L4／移行=L5〜L7）が極端に少なければ観点を見直す。重点外レンズが該当なしになるのは正常（例: 新機能モードの L5）。技法別で境界値・状態遷移が機能特性に対して0件なら穴を疑う。

## 台帳フォーマット

```markdown

- 機能略号: ORD

- レンズ別: L1×n L2×n L3×n L4×n L5×n L6×n L7×n

| ID | レンズ | 技法 | 優先度 | 観点 | 前提/手順 | 期待結果（裏側確認含む） | 関連ファイル | 状態 |
```

## 検証レンズ定義（L1〜L7）

各レンズの「疑う視点」。最低1観点を残す。すべて「壊し方の視点」で軸を揃えている（L1/L2/L5 は利用者像由来、L3/L4/L6/L7 は検証観点由来だが、いずれもレンズとして扱う）。括弧内は旧ペルソナ番号との対応。

| # | 検証レンズ | 疑う視点（代表観点） |
|---|---------|------------------|
| L1 | 直感操作で壊す（旧P1新人ユーザー） | 説明を読まず直感で操作。誤クリック・空送信・連打で壊れないか |
| L2 | 高速大量入力で壊す（旧P2ベテラン） | キーボードで高速・大量入力。Tab遷移／ショートカット／IME変換中のEnter |
| L3 | 境界・不正・権限で壊す（旧P3） | 境界値・不正値・権限外・二重送信。バリデーションや排他制御が効くか |
| L4 | 裏のDB整合を疑う（旧P4） | 画面を信用せず裏のDBテーブルを直接確認。CRUDの整合性 |
| L5 | 既存データ投入で壊す（旧P5移行担当） | 旧システムの既存データを投入。欠損・異形式・文字コード・件数の一致 |
| L6 | 周辺の回帰で壊す（旧P6） | 「今まで動いていた何か」が壊れていないか。周辺機能やF5後の挙動 |
| L7 | 仕様との不一致を暴く（旧P7） | 「実装＝正しい仕様」を信用しない。一次情報（課題・仕様書）と挙動を突合 |

## 状態定義

| 状態 | 意味 |
|---|---|
| `active` | 現仕様で有効なケース |
| `要確認` | 仕様変更で再判定が必要、または移行で旧挙動がバグ疑い。人間判断待ち |
| `obsolete` | 廃止されたケース。記録として残し削除しない |

## モード別オラクル早見表

- **新機能**: 期待結果は「あるべき仕様」に合わせる。実装の現状とズレたら実装側の不具合を疑う。
- **移行**: 期待結果は「旧システムの挙動」に合わせる。ただし旧挙動が明らかにバグなら一致させず `要確認` で人間へ。

## 制約

- 台帳に実在の顧客名・案件名・社内システムの認証情報を書かない（git 管理されるため）。
- 既存ケースを勝手に削除しない。`要確認`/`obsolete` への降格に留め、最終判断は人間に残す。
- テストコードは生成しない。コード化が必要なら `/test`・`/playwright-mcp-e2e` を案内する。
- レンズの観点漏れを黙殺しない。該当なしは必ず理由を明記する。逆に水増しの薄い1件も作らない。
- 主成果物は Markdown 台帳。CSV が必要な場合は台帳から書き出す補助エクスポートに留め、CSV を一次成果物にしない。

## 完了条件

`tests/qa/<機能名>.md` と2階層サマリ（先頭ヘッダ＋`tests/qa/README.md` index）を更新し、以下の質ゲートを満たした状態:

1. L1〜L7 すべてが観点または「該当なし＋理由」を持つ（WHO 軸の充足）。
2. 技法別カバレッジを点検し、機能特性に対する明らかな穴（例: 数値/日付を扱うのに境界値0件、状態を持つのに状態遷移0件）が無い（HOW 軸の充足）。穴があれば未完了とし観点を追加する。

コード化が必要なら `/test`・`/playwright-mcp-e2e` を案内して完了。
