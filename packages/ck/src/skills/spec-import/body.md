# spec-import

Excel・Word・PowerPoint の仕様書を、`/qa`・`/plan` が一次情報として使える**構造化 Markdown 仕様**（`specs/<機能名>.md`）に変換して保存する。`/office`（テキスト化して表示するだけ）の後工程。

**コア原則: 原文に無いことを補完しない。曖昧・欠落は「不明」として残す（そこを `/qa` の仕様突合が突く）。**

## Step 1: 入力を確定する

- `$ARGUMENTS` に仕様書ファイル（xlsx / docx / pptx）のパスが渡された場合: `Skill: ck:office`（完全修飾名）を呼び出してテキスト化する
- 直前の会話で `/office` によるテキスト化が済んでいる場合: その出力をそのまま入力に使う（再実行しない）
- どちらも無い場合はファイルパスを尋ねる（非対話実行などユーザーに質問できない場合は入力なしと報告して停止する）
- **打ち切りの確認**: `/office` はシートあたり既定200行で出力を打ち切る。取り込みは全文が前提のため、打ち切り表示があれば範囲を指定して再実行し**全文を得てから** Step 2 へ進む（打ち切られた部分を「原文に無い＝不明」と誤記しない）

## Step 2: 構造化する

テキスト化した内容を**機能単位**に分割し、各機能を以下の観点で構造化する。該当情報が原文に無い項目は省略せず「不明」と書く:

- **概要** — この機能が何をするか（1〜3文）
- **画面・入出力項目** — 名称・型・必須/任意・桁数/形式（表形式）
- **バリデーション** — 入力チェックのルールとエラー時の挙動
- **業務ルール** — 計算式・状態遷移・締め処理などのドメインロジック
- **権限** — 誰が操作できるか（記載が無ければ「不明」— 権限が絡むなら qa は full スコープになるため明記が重要）
- **要確認（不明点）** — 原文が曖昧・矛盾・欠落している箇所の一覧。**推測で埋めない**

原文の表現が曖昧な場合は言い換えずに原文を引用し、`（原文ママ）` を付けて要確認に挙げる。

## Step 3: specs/ に保存する

1. `specs/<機能名>.md`（kebab-case）に Write する。複数機能があれば機能ごとにファイルを分ける。機能名が日本語の場合は英訳またはローマ字の kebab-case に変換し（例: 「受注入力」→ `order-entry`）、対応関係をファイル先頭の取込元情報と `specs/README.md` の機能列（日本語名のまま記載）で残す。**再取込み時は README の機能列から既存ファイル名を引いて同じファイルを更新する**（変換揺れで別ファイルを作らない）
2. **既存の `specs/<機能名>.md` がある場合**（仕様書の改版再取込み等）は黙って上書きしない。既存ファイルを Read して要確認の消化状況を確認し、「改版で置き換える（未解決の要確認は新版に引き継ぐ）」旨をユーザーに確認してから更新する（非対話時は置き換えず、新版案の提示までで停止する）
3. ファイル先頭に取込元情報を記録する（取込日は `date +%Y-%m-%d` で取得する）:

```markdown
# <機能名> 仕様
- 取込元: <元ファイル名>（シート/章: <該当箇所>）
- 取込日: YYYY-MM-DD
- 要確認: N件
```

4. 横断 index `specs/README.md`（機能・ファイル・取込元・取込日・要確認件数の表）を更新する。無ければ新規作成する

## Step 4: 伏字化の確認

`specs/` は git 管理される。実在の顧客名・住所・電話番号・メール・認証情報を検知したら、そのまま書かず伏字化（例: `<顧客A>`・`<TOKEN>`）を提案し、ユーザー承認後にのみ記録する。仕様書のサンプルデータ欄は実データであることが多いので特に注意する。

## Step 5: 次工程を案内する

1. ステージング案内を出力する:
   > 仕様を `specs/<機能名>.md` に取り込みました。ステージングの準備ができたら `git add specs/` を実行してください。
2. 次工程を案内する:
   - テストケース設計へ進むなら `/qa <機能名>` を案内する（Step 2 の一次情報として `specs/<機能名>.md` を指定できる。要確認一覧は仕様突合の入力になる）
   - 実装計画へ進むなら `/plan` を案内する

## 制約

- 原文に無い仕様を創作しない。曖昧さの解消はユーザー・顧客への確認事項として「要確認」に残す
- 元ファイル（xlsx / docx / pptx）を変更しない
- 書き込み先は `specs/` のみ（`specs/README.md` 含む）
- `git add` を自動実行しない。記録後にステージング案内を出力する

## 完了条件

`specs/<機能名>.md` と `specs/README.md` index の更新が終わり、機能一覧・要確認件数を報告して次工程（`/qa`・`/plan`）を案内した時点で完了。
