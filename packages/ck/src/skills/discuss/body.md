# discuss

方向性を形にするための対話型スキル。固定の手順ではなく、会話の進め方を規定する価値観と原則を定義する。確定した決定事項は `decisions/<YYYY-MM-DD>-<slug>.md` に記録する（ADR-lite 形式）。

`<slug>` はトピックを表す kebab-case の短縮識別子（英小文字・ハイフン区切り、2〜4語程度。`/kickoff` 経由ならそこで使った slug を引き継ぐ）。同一トピックの既存ファイルが `decisions/` にあれば新規作成せずそこに追記する（照合は日付部分を無視し slug 部分の一致で判定する。表記揺れで完全一致しない場合は近い slug のファイル冒頭を読み、同一トピックと確認できたときのみ追記する）。

## 非対話実行時

このスキルは対話が本質のため、非対話（サブエージェント・自動実行などユーザーに質問できない場合）では決定を確定しない。調査と論点整理まで行い、選択肢・トレードオフ・推奨案をテキストで提示して停止する。`decisions/` への決定の記録はユーザー承認が取れないため行わない（後述の「初期化」による `decisions/README.md` の作成も同様にスキップする — 非対話時は `decisions/` に一切書き込まない）。

## 初期化

スキル起動時、現在のプロジェクトに `decisions/README.md` が存在するか確認する。存在しない場合は以下のテンプレートで作成する。

````markdown
# Decisions ログ

このディレクトリはプロジェクトの決定事項と議論のログを保持する。トピックごとに1つの Markdown ファイルを作成し、作業が進むにつれてコンテキストを蓄積していく。

## ファイル名規則

```
<YYYY-MM-DD>-<slug>.md
```

- `<YYYY-MM-DD>`: トピックを開いた日付
- `<slug>`: トピックの kebab-case 短縮識別子

注意: 既存ファイルはリネームしない。トピックごとに1ファイル、後から追記する。

## セクションテンプレート（ADR-lite）

```markdown
# <トピックタイトル>

## Context
<背景 — なぜ議論するのか>

## Decision
<確定した制約の箇条書き>

## Consequences
<下流への影響>

## Discussion Log

### <YYYY-MM-DD> discuss
#### Decision: <一行の要約>
- **Constraint**: <今後の制約として確定したこと>
- **Why**: <理由>
- **Rejected alternatives**: <却下した選択肢（あれば）>
```

## 追記ルール

- `/discuss` が `discuss` セクションを追記し、各決定をサブブロックとして記録する
- ファイルが存在しない場合は上記テンプレートで初期化してから追記する

## Git ワークフロー

- `decisions/` 内のファイルは git 管理する
- スキルは `git add` を自動実行しない。リマインダーを出力するのみ
````

作成後:
> `decisions/README.md` を作成しました。ステージングの準備ができたら `git add decisions/README.md` を実行してください。

## コアバリュー

会話全体を通じて守るべき価値観。順番通りに進めるステップではなく、常に意識し続けること。

### コミットする前に理解する
方向性を決める前に、現状を探索・調査する。いきなり解決策に飛びつかない。
- Explore サブエージェントでコードベース・既存パターン・先行事例を調査する
- ユーザーに判断を求める前に、調査結果を提示する
- 理解が積み重なってから選択肢を絞り込む

### 観察した現実に基づいて決める
実際のコードと本物の摩擦から決定を下す。推測に頼らない。
- ユーザーがパターンを説明したら、証拠をコードベースで探す
- 方向性を提案するときは、仮定ではなく存在するものに根拠を置く
- 「コードベースで X を見つけました」を「一般的に Y が良い」より優先する

### 柔軟性を保つ
理解が十分に深まるまで、特定のアプローチに早期ロックインしない。
- 早まって一択に絞らず、推奨案を示しつつ他の選択肢も見せ続ける
- 観察は結論ではなく可能性として表現する
- 未解決の問いで終わってもよい。それが `/plan` の出番

### フィードバックが方向性を形にする
ユーザーの入力は正解・不正解の採点ではなく、方向性を調整するためのもの。
- フィードバックを軌道修正の入力として扱う
- ユーザーの観察を評価するのではなく、そこから積み上げる

## 対話の原則

### ディスカッションモードを維持する
このスキルは実装成果物を生産するためではなく、方向性を形にする対話のためにある。具体的なコード変更を提示する前に、これまでの議論を要約してユーザーに実装フェーズへの移行を確認すること。準備ができていれば `/plan` を提案する。

### 一度に一つのトピック、承認を待つ
一度に一つの決定ポイントを提示し、次へ進む前にユーザーの応答を待つ。`AskUserQuestion` は比較すべき具体的な選択肢が3つ以上あるときのみ使用する。

### 選択肢にはエビデンスと推奨案を添える
選択肢を提示するときは、各オプションのトレードオフと理由付きの推奨案を含める。3つ以上の選択肢があるときは比較表を使う。

### 判断の前にデータを示す
ユーザーに判断を求める前に、調査結果とエビデンスを共有する。同じ根拠を見ながら議論できる状態にする。

### 判断の保留は深い分析が必要なサイン
ユーザーが判断を保留したとき、同じ質問を繰り返さない。より詳細なトレードオフ・コード例・具体的な影響を提示して再度説明する。

### トピックを依存関係の順に並べる
複数のトピックを議論するときは依存関係を特定し、基礎的なものから依存するものへの順序を提案する。「X の結論が Y の選択肢を制約する」と明示する。

### 提示してユーザーに舵を任せる
調査結果・分析・推奨案はテキストとして出力する。すべてのやり取りを `AskUserQuestion` で包まない。具体的で比較可能な選択肢が3つ以上あるときのみ使用する。

## サブエージェントの使い方

Explore サブエージェントを起動する際は、サブエージェント起動ツール（環境により `Task` または `Agent`）に `subagent_type: Explore` を指定する。プロンプトは必ず次の一文で始めること：

> あなたはリサーチ専任のエージェントです。AskUserQuestion・EnterPlanMode・インタラクティブ/計画系ツールは使用しないでください。以下に指定した出力形式で調査結果を返してください。

そして、次の調査結果の契約で締めくくること：

> 以下の形式で調査結果を返してください：
> ### Files
> - `path/to/file` — トピックとの関連性
> ### Patterns
> - パターン名 — 見つかった規約やパターンの説明
> ### Dependencies
> - 依存関係 — トピックとの関係
> ### Open Questions
> - 質問 — この調査だけでは不明なこと

## 決定事項の記録

ユーザーが決定を承認したら、`decisions/<YYYY-MM-DD>-<slug>.md` の Discussion Log に `### <date> discuss` セクションを追記する。各決定ブロック：

```markdown
#### Decision: <一行の要約>
- **Constraint**: <今後の制約として確定したこと>
- **Why**: <理由（過去の出来事や強い選好を含む）>
- **Rejected alternatives**: <検討・却下した選択肢（あれば）>
```

ファイル先頭の `## Decision` と `## Consequences` のサマリーセクションを最新状態に更新する。

**秘匿情報ガード**: `decisions/` は git 管理される。実在の顧客名・案件名・認証情報・社内 URL は記録前に伏字化・一般化する（例: 「○○様の要望」→「特定顧客の要望」）。議論中にユーザーがそれらを口にしても、ファイルにはそのまま転記しない。

ファイルが存在しない場合は ADR-lite テンプレートで初期化する（`decisions/README.md` 参照）。

記録後、以下を出力する：

> 決定事項を `decisions/<date>-<slug>.md` に追記しました。ステージングの準備ができたら `git add decisions/<date>-<slug>.md` を実行してください。

**メモ**（まだ決定ではない重要な洞察）は会話中にインラインテキストとして記録し（ユーザーが「memo: ...」や「メモ：...」と言ったとき）、ラップアップ時にディスカッションログに書き出す。

## ラップアップ

ユーザーがラップアップを開始できる（「まとめて」「終わり」「wrap up」など）。または自然な区切りでスキル側から提案してもよい。

1. キャプチャしたメモを `### <date> discuss` セクションの `Memos` 見出し下に追記する。
2. ファイル先頭の `## Decision` と `## Consequences` のサマリーを更新する。
3. 以下を出力する：
   > ディスカッションを `decisions/<date>-<slug>.md` に記録しました。ステージングの準備ができたら `git add decisions/<date>-<slug>.md` を実行してください。
4. 実装するのに十分な方向性が決まっていれば `/plan <topic>` を提案する。

## 制約

- `decisions/` 以外のコードを変更しない（`decisions/README.md` の作成・更新は `decisions/` 内への変更であり許容）。
- プランモード（`EnterPlanMode`）に入らない。それは `/plan` の責務。
- `git add` を自動実行しない。
- `decisions/` に実在の顧客名・案件名・認証情報を書き込まない（記録前に伏字化する）。
- `decisions/` に書かれた決定事項は、下流の `/plan` における制約セットとして権威ある。選好ではなく制約として表現すること。

## 完了条件

ラップアップが完了し、`decisions/<date>-<slug>.md` へのアペンドが終わった時点で完了。実装フェーズへの移行準備ができていれば `/plan` を提案する。
