# test

実装完了後にユニットテストを作成・実行する。E2E は必要性をユーザーに確認してから `playwright-mcp-e2e` スキルに委譲する。

**位置づけ**: 実装完了 → `/test` → `/review` → `/pr`

## Step 1: 変更ファイルを把握する

```bash
git diff --name-only HEAD~1 HEAD 2>/dev/null || git diff --name-only
```

変更ファイルを確認し、テスト対象となる関数・クラス・モジュールを洗い出す。

## Step 2: テストフレームワークを検出する

プロジェクトの `package.json` を確認して使用中のフレームワークを特定する:

| 検出キー | フレームワーク | 実行コマンド例 |
|---------|--------------|--------------|
| `vitest` | Vitest | `npx vitest run` |
| `jest` | Jest | `npx jest` |
| `bun test` | Bun test | `bun test` |
| `pytest` | pytest | `pytest` |
| `go test` | Go test | `go test ./...` |

テストコマンドが `package.json` の `scripts` にある場合はそちらを優先する。

## Step 3: 既存テストを確認する

変更ファイルに対応するテストファイルが既に存在するか確認する:
- `*.test.ts` / `*.spec.ts`
- `__tests__/` ディレクトリ
- 既存テストがあれば内容を読み、追加すべきケースを特定する

## Step 4: ユニットテストを作成する（必須）

以下を網羅するテストを書く:

**必須カバレッジ:**
- [ ] 正常系（期待通りの入力と出力）
- [ ] 境界値（空配列・null・0・最大値など）
- [ ] 異常系（不正な入力・エラーケース）

**テスト作成の原則:**
- 既存テストのスタイル・命名規則に従う（見つかった場合）
- モックは外部依存（DB・API・ファイルシステム）のみに限定する
- テスト名は「〜のとき、〜になる」形式で意図を明示する
- 1テスト1アサーションを基本とする

テストファイルの配置は既存プロジェクトの規約に従う。規約が不明な場合は対象ファイルと同じディレクトリに `<name>.test.ts` を作成する。

## Step 5: テストを実行して確認する

Step 2 で特定したコマンドでテストを実行する。

**失敗した場合:**
- エラーメッセージを読み、テストコードまたは実装コードの問題を特定する
- 修正して再実行する
- 修正が実装コードに及ぶ場合はユーザーに報告してから変更する

**全テスト通過を確認してから次へ進む。**

## Step 6: E2E テストの要否を確認する（任意）

`AskUserQuestion` で確認する:

- 「追加する（ブラウザ操作が必要な機能）」
- 「スキップする（APIのみ・ロジックのみ）」

「追加する」を選んだ場合は `playwright-mcp-e2e` スキルを呼び出す:
```
Skill: playwright-mcp-e2e
```

## Step 7: 完了報告

```
## テスト結果
- ユニットテスト: ✅ N件 作成・全通過
- E2E テスト: ✅ 作成済み / ⏭️ スキップ

次のステップ: `/review`
```

## 制約

- テストが1件でも失敗している状態で完了を宣言しない
- 既存テストを削除・修正するときはユーザーに理由を説明してから行う
- テスト通過のためだけに実装コードを歪めない（テストに合わせてロジックを正しく直すのはOK）

## 完了条件

Step 7 の完了報告を出力し、全ユニットテストが通過している状態で完了。次は `/review` を案内する。
