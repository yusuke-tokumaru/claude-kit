# test

実装完了後にユニットテストを作成・実行する。E2E は必要性をユーザーに確認してから `playwright-mcp-e2e` スキルに委譲する。

**位置づけ**: 実装完了 → `/test` → `/review` → `/pr`。`/qa` で `tests/qa/<機能>.md` 台帳を設計済みなら、その `active` ケースをコード化する後段としても働く。

## Step 1: 変更ファイルを把握する

```bash
# デフォルトブランチとの分岐点からの全変更を対象にする（複数コミットでも漏れない）
BASE=$(git symbolic-ref refs/remotes/origin/HEAD 2>/dev/null | sed 's|refs/remotes/origin/||')
git diff --name-only "origin/${BASE:-main}...HEAD" 2>/dev/null \
  || git diff --name-only HEAD~1 HEAD 2>/dev/null \
  || git diff --name-only
```

未コミットの変更がある場合は `git diff --name-only` の結果も対象に加える。

変更ファイルを確認し、テスト対象となる関数・クラス・モジュールを洗い出す。

## Step 2: テストフレームワークを検出する

プロジェクトの `package.json` を確認して使用中のフレームワークを特定する:

| 検出キー | フレームワーク | 実行コマンド例 |
|---------|--------------|--------------|
| `vitest` | Vitest | `npx vitest run` |
| `jest` | Jest | `npx jest` |
| `bun test` | Bun test | `bun test` |
| `pytest` | pytest | `pytest` |
| `go test` | Go test | `go test ./...` |

テストコマンドが `package.json` の `scripts` にある場合はそちらを優先する。

## Step 3: 既存テストを確認する

変更ファイルに対応するテストファイルが既に存在するか確認する:
- `*.test.ts` / `*.spec.ts`
- `__tests__/` ディレクトリ
- 既存テストがあれば内容を読み、追加すべきケースを特定する

## Step 3.5: QA 台帳を参照する（あれば）

`/qa` が設計した台帳がある場合は、それをテスト対象の一次ソースにする。Step 1 で洗い出した機能名から `tests/qa/<機能>.md` を探し、無ければ `tests/qa/README.md`（横断 index）から関連台帳を引く。

- 存在する場合は読み込み、`状態: active` のケースを**ユニットテストで再現可能なもの**に絞ってテスト対象に変換する（観点・前提/手順・期待結果〈裏側確認含む〉をそのままテストの意図とアサーションに写す）。台帳のケースID（例 `ORD-C1`）をテスト名またはコメントに残し、台帳とコードを相互追跡できるようにする。
- `状態: 要確認` のケースは人間判断待ちなのでテスト化せず、ユーザーに「要確認 N 件は台帳側で解決してからテスト化します」と伝える。`obsolete` は対象外。
- ブラウザ操作前提のケース（L1/L2 の UI 操作など）はユニットでは再現できないため Step 6 の E2E 候補として控える。
- 台帳が存在しない場合はこのステップをスキップし、変更ファイルから直接テスト対象を起こす（Step 4 に進む）。

## Step 4: ユニットテストを作成する（必須）

Step 3.5 で台帳ケースが得られた場合はそれを起点に、無い場合は変更ファイルから、以下を網羅するテストを書く:

**必須カバレッジ:**
- [ ] 正常系（期待通りの入力と出力）
- [ ] 境界値（空配列・null・0・最大値など）
- [ ] 異常系（不正な入力・エラーケース）

**テスト作成の原則:**
- 既存テストのスタイル・命名規則に従う（見つかった場合）
- モックは外部依存（DB・API・ファイルシステム）のみに限定する
- テスト名は「〜のとき、〜になる」形式で意図を明示する
- 1テスト1アサーションを基本とする

テストファイルの配置は既存プロジェクトの規約に従う。規約が不明な場合は対象ファイルと同じディレクトリに `<name>.test.ts` を作成する。

## Step 5: テストを実行して確認する

Step 2 で特定したコマンドでテストを実行する。

**失敗した場合:**
- エラーメッセージを読み、テストコードまたは実装コードの問題を特定する
- 修正して再実行する
- 修正が実装コードに及ぶ場合はユーザーに報告してから変更する

**全テスト通過を確認してから次へ進む。**

## Step 6: E2E テストの要否を確認する（任意）

`AskUserQuestion` で確認する:

- 「追加する（ブラウザ操作が必要な機能）」
- 「スキップする（APIのみ・ロジックのみ）」

「追加する」を選んだ場合は `playwright-mcp-e2e` スキルを呼び出す:
```
Skill: playwright-mcp-e2e
```

## Step 7: 完了報告

```
## テスト結果
- ユニットテスト: ✅ N件 作成・全通過
- E2E テスト: ✅ 作成済み / ⏭️ スキップ

次のステップ: `/review`
```

## 制約

- テストが1件でも失敗している状態で完了を宣言しない
- 既存テストを削除・修正するときはユーザーに理由を説明してから行う
- テスト通過のためだけに実装コードを歪めない（テストに合わせてロジックを正しく直すのはOK）

## 完了条件

Step 7 の完了報告を出力し、全ユニットテストが通過している状態で完了。次は `/review` を案内する。
