# plan

コンテキスト圧縮を乗り越える実装計画を生成する。`decisions/<YYYY-MM-DD>-<slug>.md` から適用可能な決定事項を読み込み、コードベースを調査し、設計選択を提示して、プランモードでステップバイステップの計画を確定する。

**非対話実行（サブエージェント・自動実行などユーザーに質問できない場合）**: 「ユーザーに確認」「`AskUserQuestion`」のステップは推奨案を採用して進める（Step 2 は「現在のブランチで続ける」を既定とする）。`EnterPlanMode` / `ExitPlanMode` が使えない場合はプランファイルの作成と内容の提示で代替し、Step 5 の議論項目は「推奨案＋根拠」をプランに記録して先へ進む。置いた仮定は最終報告で「本来ユーザーに確認すべきだった点」として列挙する。

## 手順

### Step 1: 調査

- **決定事項の読み込み**: `decisions/` でトピックのスラグに一致するファイルを検索する。見つかった場合は `## Decision` セクション、および `/debug` が記録した `### <date> debug` ブロック（`#### Root cause` / `#### Fix direction`）を**必須制約**として扱う。すべての適用可能な決定はプランのステップに追跡可能でなければならない。却下された代替案は除外制約として扱う。
- **コードベース調査**: 以下の制約ブロックと出力契約を使って Explore サブエージェント（起動ツールは環境により `Task` または `Agent`、`subagent_type: Explore`）を1つ起動する:
  > あなたはリサーチ専任のエージェントです。AskUserQuestion・EnterPlanMode・インタラクティブ/計画系ツールは使用しないでください。以下に指定した出力形式で調査結果を返してください。
  >
  > ### Files
  > - `path/to/file` — タスクとの関連性
  > ### Patterns
  > - パターン名 — 見つかった規約やパターンの説明
  > ### Dependencies
  > - 依存関係 — タスクへの影響
  > ### Open Questions
  > - 質問 — この調査だけでは不明なこと
- **外部調査**（外部ライブラリ・サービスが関係する場合のみ）: 外部調査サブエージェントを並列で起動する。
- すべての調査結果を Common Context ブロックにまとめる。調査によって根本的に異なるアプローチが明らかになった場合は、先に進む前にユーザーに選択を求める（`AskUserQuestion` を使用）。

### Step 2: ブランチ戦略

`AskUserQuestion` で2つの選択肢を提示する:
- 「新しいブランチを作成する」
- 「現在のブランチで続ける」

ユーザーが確認するまで先に進まない（非対話時は冒頭の規定に従い「現在のブランチで続ける」を採用する）。

「新しいブランチを作成する」が選ばれた場合は、リポジトリの命名慣行（`git branch -a` で確認できる既存の型）に沿ってブランチ名を確定し、Step 3 の「タスクの要約とブランチ名」に記録する。**ブランチの作成・切替は plan では行わない**。implement が着手時にプラン記載のブランチ名で作成・切替する。

### Step 3: プランを作成する

Markdown でプランコンテンツを作成する。必須要素:

- **タスクの要約とブランチ名**
- **調査サマリー** — 主な調査結果、影響を受ける領域、関連パターン
- **反映された設計決定** — 読み込んだ各決定をそれを実装するステップにマッピングした表
- **ステップ間共有コンテキスト** — 複数のステップから参照される型・定数・契約
- **ステップ** — 各ステップに含めるもの:
  - 進捗マーカー `- [ ]`（実装中に `- [x]` に更新）
  - コミットメッセージ（命令形）
  - フルパスと理由を含むファイル変更
  - 検証基準（テストコマンド、期待される動作、手動確認）
  - 先行ステップへの依存関係と、このステップが生産するもの
- **リスクと軽減策** — 具体的な軽減策を伴う少なくとも1つのリスク

条件付き要素:
- **議論が必要な設計決定** — 選択肢・トレードオフ・推奨案を含むラベル付きコールアウトとして提示。コールアウトが必要な例: アーキテクチャパターン、ライブラリ/フレームワーク選定、データモデル、API契約、状態管理、認証戦略。不要な例: 変数名、ヘルパー構造、すでに決定済みアーキテクチャ内のファイル構成。
- **曖昧さ** — 不足情報に対するラベル付きコールアウト。

### Step 4: プランモードに入る

`EnterPlanMode` を呼び出す。プランファイルのパスは `plans/<YYYY-MM-DD>-<slug>.md` とし、Write ツールでプランコンテンツを書き込む（プランモードのシステムメッセージで別途パスが指定された場合はそちらに従う）。

### Step 5: プラン議論フェーズ

議論項目（レビューポイント＋設計決定）を抽出する。`/discuss` と同じ対話原則に従って一つずつ提示する:
- コンテキストをテキスト出力として提示し、ユーザーが自由に応答できるようにする
- 次の項目に進む前にユーザーの応答を待つ
- 設計決定にはトレードオフと推奨案を提示する
- ユーザーが判断を保留したら、より深い分析を提供してから再提示する

各結論をプランに即時反映する:
- レビューポイントの結論: `**User Context**: <訂正または追加情報>`
- 設計決定の結論: コールアウトを確定した選択肢に更新する

### Step 6: プランモードを抜ける

すべての議論項目が解決したら（またはユーザーが準備完了を示したら）`ExitPlanMode` を呼び出す。ユーザーがプランをレビューして承認する。

### Step 7: decisions/ に追記する

`decisions/<date>-<slug>.md` に `### <YYYY-MM-DD> plan` セクションを追記し、短い要約とプランファイルへのパスを記録する。以下を出力する:

> プランのサマリーを `decisions/<date>-<slug>.md` に追記しました。ステージングの準備ができたら `git add decisions/<date>-<slug>.md` を実行してください。

### Step 8: 実行を implement に委譲する

**このスキルは計画専任。実装そのものは行わない。** プラン承認後は `Skill: ck:implement`（プラグイン名前空間付きの完全修飾名）を呼び出して実行フェーズを委譲する。implement が承認済みプラン（`plans/<date>-<slug>.md`）のステップを順に実装し、各ステップの検証基準（テストコマンド・期待動作）を満たしながら進捗マーカーを `- [ ]` → `- [x]` に更新する。implement の完了後に `/test` → `/review` のチェーンへ進む。

## コンテキスト圧縮耐性ガイドライン

プランはコンテキスト圧縮を乗り越えなければならない。

- **明示的な依存チェーン**: 各ステップが何に依存し、何を生産するかを記述する。ステップの順番だけに頼らない。
- **散文よりコード**: 散文の説明よりコードスニペットと構造化データを優先する。コードは intact で残る。散文は要約される。
- **ラベル付きコールアウト**: 設計の根拠を `**Why**: ...` コールアウトとして記述し、段落に埋め込まない。
- **読み込んだ決定事項はIDで参照**: ステップが `decisions/` の決定を実装する場合、ファイルと決定タイトルを参照する。

## スキルの境界

- 実装はこのスキルで行わない。承認済みプランの実行は `implement` に委譲する（ユーザーが `ExitPlanMode` で承認するまで委譲しない）。
- 設計決定やレビューポイントが存在する場合、プラン議論フェーズをスキップしない。
- `git add` を自動実行しない。
- 各ステップはコミット・検証基準・明示的な依存関係を定義しなければならない。
- プランは具体的な軽減策を伴う少なくとも1つのリスクを特定しなければならない。
- `decisions/` から読み込んだすべての適用可能な決定はプランのステップに反映されなければならない。却下された代替案を再提案してはならない。

## 完了条件

Step 7 まで完了し、承認済みプランを `implement` に引き渡した時点で完了。実装の進捗（全マーカー `- [x]`）と検証は implement 側の責務であり、その完了後に `/test` → `/review` へ進む。
