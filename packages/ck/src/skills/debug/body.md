# debug

バグ・エラー・予期しない挙動の根本原因を**証拠で**特定し、`decisions/` に記録してから修正方針を合意するスキル。

**コア原則: コード側証拠と実行時証拠の両方が揃うまで原因を断言しない。原因を断言するまで修正を始めない。**

**位置づけ**: `/kickoff` がバグをここにルーティングする。修正完了後は `/test` → `/review` の既存チェーンへ。

## 前提

本スキルは体系的デバッグの規律を内蔵しており、単独で完結する。常に以下を守る:

- **Iron Law（証拠優先）**: コード側証拠と実行時証拠の両方が揃うまで原因を断言せず、断言するまで修正に着手しない。
- **仮説は1つずつ最小検証**: 一度に複数を変えない。否定した仮説は `Ruled out` として控える。
- **3回失敗でアーキテクチャを疑う**: 同じ修正の試行が3回失敗したら4回目を試さず、設計の問題を疑って `Ruled out` を見返しユーザーと議論する（Phase 2・Red Flags 参照）。

**非対話実行（サブエージェント・自動実行などユーザーに質問できない場合）**: 「ユーザーに確認」「合意したら」のステップは、推奨案を採用して進めてよい。ただし置いた仮定を decisions の記録に `Assumptions` として残し、最終報告で「本来ユーザーに確認すべきだった点」を列挙すること。decisions 記録前に修正しない等の制約は非対話でも緩和されない。

## Phase 1: 症状の把握と再現

1. エラーメッセージ・症状・再現手順をユーザーから確認する。スタックトレースは最後まで読む
2. バグのスラグを決める（`/kickoff` と同じ規則。`#42` を含む場合は `42-fix-login` のように Issue 番号プレフィックス）
3. **再現を試みる**（コマンド実行・テスト実行・ブラウザ操作）。再現できない場合は推測で進めず、追加情報（環境・ログ・タイミング）をユーザーに求める
4. `git log --oneline -10` と `git diff` で最近の変更を確認する

## Phase 2: 証拠収集（UI / 非 UI で分岐）

仮説は**1つずつ**検証する。否定した仮説は後で `Ruled out` として記録するため控えておく。

### UI バグ（ブラウザ上の症状）の場合

**Playwright MCP による視覚確認を必須とする。コードを読んで原因がわかった気がしてもスキップしない。**

```
browser_navigate(<該当URL>)
  → browser_snapshot()                  # DOM構造の確認
  → 操作を再現（click / type / press_key）
  → browser_take_screenshot()           # 視覚的状態の記録
  → browser_console_messages()          # JSエラーの取得
```

- 開発サーバーが起動していなければ起動方法を案内してから進む
- ツールの詳細・トラブルシューティングは `playwright-mcp-e2e` スキルを参照する

### 非 UI バグ（CLI / API / ロジック / ビルド）の場合

実行時証拠 = 再現コマンドの出力・失敗するテスト・ログ。

- 多層システム（CI → ビルド → 実行、API → サービス → DB など）では、**修正案を出す前に**各層の境界で入出力をログし、どの層で壊れているかを特定する
- エラーがコールスタックの深部で起きている場合は、不正な値の発生源まで遡る（症状の場所ではなく発生源を疑う）

## Phase 3: 原因の断言と記録

### 報告形式

```
【原因】<1行で断言>
【証拠】
- コード側: `path/to/file:行番号` — <問題の説明>
- 実行時: <再現出力・失敗テスト・コンソールエラー（UI ならスクリーンショット参照）>
【影響範囲】<関連する機能・画面・モジュール>
```

**両方の証拠が揃わない場合は「原因不明」とし、Phase 2 に戻る。** 推測のまま Phase 4 に進まない。

### decisions/ への記録（修正前に必ず実行）

`decisions/<YYYY-MM-DD>-<bug-slug>.md` に追記する。ファイルが無ければ ADR-lite テンプレートで初期化する（`decisions/README.md` があれば従い、無ければ以下のブロックで十分）:

```markdown
### <YYYY-MM-DD> debug

#### Root cause: <一行で断言>
- **Evidence (code)**: `path/to/file:行番号` — <問題の説明>
- **Evidence (runtime)**: <再現コマンド・失敗テスト・コンソールエラー>
- **Impact**: <影響範囲>
- **Ruled out**: <調査で否定した仮説（あれば。次セッションの重複調査を防ぐ）>
```

記録後:
> 調査結果を `decisions/<date>-<bug-slug>.md` に追記しました。ステージングの準備ができたら `git add decisions/<date>-<bug-slug>.md` を実行してください。

## Phase 4: 修正方針の合意と出口

1. 修正方針を2〜3案、トレードオフ（工数・影響範囲・リスク）と推奨案つきで提示する（`AskUserQuestion` は比較可能な3案以上のときのみ）
2. ユーザーが合意したら `Fix direction` ブロックを decisions に追記する:

```markdown
#### Fix direction: <合意した修正方針>
- **Why**: <理由>
- **Rejected fixes**: <却下した修正案（あれば）>
```

3. **修正規模をユーザーに確認して出口を分岐する:**

| 規模 | 出口 |
|------|------|
| 軽微（影響1〜2ファイル・方針が自明） | このスキル内で完結: 失敗するテストを先に作成 → 最小修正 → テスト通過を検証。完了後 `/test` → `/review` を案内 |
| 構造的（複数モジュール・設計判断を含む） | `/plan <bug-slug>` を案内して終了。記録済みの `Fix direction` が /plan の必須制約として読み込まれる |

## Red Flags — STOP

これらの思考が浮かんだら止まる:

- 「コードを見れば分かる」→ UI バグの視覚確認はスキップ不可
- 「とりあえず直してみる」→ Phase 3 の原因断言前の修正は禁止
- 「複数まとめて変えて様子を見る」→ 仮説は1つずつ。どれが効いたか分からなくなる
- 「ついでにリファクタリングも」→ 修正は最小限。バンドル禁止
- **修正の試行が3回失敗** → 4回目を試さない。アーキテクチャの問題を疑い、`Ruled out` を見返してユーザーと議論する

## 制約

- Phase 3 の decisions 記録前にコードを修正しない
- 再現できないバグを推測で修正しない
- 既存テストの削除・改変で「修正」しない
- `git add` を自動実行しない

## 完了条件

- 軽微修正の場合: 修正がテストで検証済み・decisions 記録済みの状態で `/test` を案内した時点
- /plan 委譲の場合: `Fix direction` 記録済みの状態で `/plan <bug-slug>` を案内した時点
