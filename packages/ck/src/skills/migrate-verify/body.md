# migrate-verify

移行データの**旧新突合（リコンサイル）を機械的に実行する**。`/qa`（移行モード）が設計した「何を確認すべきか」（L4・L5 ケース）に対し、実データで「実際に一致しているか」を検証する実行系。

**位置づけ**: `/qa` 移行モードの後段または並走。突合結果は qa 台帳のケース状態に反映する。台帳が無くても単独で起動できる。

**コア原則: 移行データに書き込まない（突合は読み取り専用）。不一致の正否を断定しない（オラクル判断は人間に残す）。**

## 非対話実行時

突合条件の確認（Step 1）は一次情報からの推定を採用して進めてよい。ただし推定した条件（結合キー・許容差・データ入手方法）をレポートと最終報告に仮定として明記する。読み取り専用の制約は非対話でも緩和しない。Step 2 の「提示してから実行」ができないため、生成したスクリプトが SELECT のみであることを自身で確認できた場合に限り実行してよい（一時テーブル等の承認を要する操作が必要なら、スクリプトの提示までで停止する）。接続情報が指定されていない場合はプロジェクトの設定（`.env*`・設定ファイル・README の接続手順）から読み取り、見つからなければ Step 3 の「実行できない環境」分岐（スクリプト提示までで停止）に従う。

## Step 1: 対象と突合条件を確定する

- `$ARGUMENTS` から対象（機能名・テーブル・ファイル）を特定する
- 一次情報を集める: `tests/qa/<機能名>.md`（移行モード台帳。**あれば** — 無くても単独起動できる）の L4・L5 ケース・旧新スキーマ・マッピング定義
- 突合条件を確定する: **結合キー**（旧新レコードを対応付ける列）・**比較列**・**許容差**（丸め・時刻精度・文字正規化など意図的な差）
- データの入手方法（DB 接続 / ダンプ / CSV）や許容差が曖昧な場合は `AskUserQuestion` で確認する

## Step 2: 突合スクリプトを生成する

以下の4種を基本セットとして、環境に合わせた形（SQL・スクリプト等）で生成する:

| 種別 | 何を突合するか |
|------|----------------|
| 件数突合 | 全件数と、グループ別件数（月別・種別・ステータス別など）の一致 |
| キー突合 | 旧にあって新にないキー ／ 新にあって旧にないキー |
| 値突合 | 比較列の不一致行（キー単位）。金額・数量は集計値（SUM）でも突合する |
| 形式突合 | NULL・空文字・文字コード・日付形式の変換もれ（L4 の代表観点） |

- **読み取り専用で書く**（SELECT のみ。旧新どちらにも書き込まない）。一時テーブル等が必要な場合はユーザーの明示承認を得てから使う
- 生成したスクリプトはユーザーに提示してから実行する（接続先の確認を兼ねる）

## Step 3: 実行して不一致を分析する

- 実行結果から不一致を「**移行バグ疑い**／**旧データ起因**（旧側の汚れ）／**仕様差**（意図的な変換）」に仮分類する
- 断定せず、各分類に根拠（件数・代表キーの例）を添える
- 実行できない環境（接続情報が無い等）の場合は、スクリプトと実行手順の提示までで止め、その旨を報告する

## Step 4: 結果を記録する

1. レポートを `tests/qa/recon/<YYYY-MM-DD>-<機能名>.md` に以下の形式で Write する（構造を固定して実行ごとのぶれを防ぐ）:

```markdown
# <機能名> 突合レポート
- 対象: <テーブル・ファイル>
- 突合条件: 結合キー=<列> / 比較列=<列…> / 許容差=<丸め・時刻精度など>
- 仮定: <推定した条件。無ければ「なし」>

## 結果サマリ
| 種別 | 結果 | 不一致件数 |
|------|------|-----------|
| 件数突合 | 一致 / 不一致 | N |
| キー突合 | 一致 / 不一致 | N |
| 値突合   | 一致 / 不一致 | N |
| 形式突合 | 一致 / 不一致 | N |

## 不一致詳細
- <種別> / <代表キー（伏字化）> / <概要> / 仮分類: 移行バグ疑い | 旧データ起因 | 仕様差

## 台帳への反映
- <ケースID> → 要確認（理由タグ=突合） / 対応ケースなし→新規起票
```

2. `tests/qa/<機能名>.md` 台帳があれば、対応する L4・L5 ケースに結果を反映する: 一致したケースはそのまま、不一致が出たケースは `要確認`（理由タグ=突合）に降格し、観点欄に不一致の概要を追記する。**引き当て方**: 各ケースの観点・関連ファイル欄を読み、不一致の種別と突き合わせる（形式・値の変換もれ→L4 系、件数・キー・集計のずれ→L5 系が典型）。対応するケースが特定できない不一致は `/qa` の起票規約（Step 4.7）に従い新規 `要確認` ケースとして起票する: ID は既存連番の次、レンズ列は L4/L5 のうち近い方を主分類に、技法・優先度も埋める（勝手に既存ケースへ相乗りさせない）。台帳が無い単独起動時はレポートの「## 台帳への反映」に「対応台帳なし（`/qa` 移行モードでの台帳化を推奨）」と記載する
3. ケースを降格・起票した場合は、台帳ヘッダの集計（ケース総数の状態別内訳・要確認件数・最終更新。**起票時はレンズ別・技法別カバレッジも**）と横断 index `tests/qa/README.md` の該当行（要確認件数・更新日）も併せて更新する（`/qa` が維持する2階層サマリと「レンズ別合計=現役ケース総数」の不変条件を狂わせない）
4. `git add` は自動実行しない。記録後にステージング案内を出力する

## 制約

- 移行データ（旧・新とも）に書き込まない。突合クエリは SELECT のみ
- レポート・台帳は git 管理される。実データの顧客名・住所・電話番号・メール・認証情報をそのまま転記しない（キー値は伏字化し、件数と代表例のみ記録する）
- 不一致の正否を断定しない。人間判断は qa の `要確認` フローに回す
- 台帳ケースを削除しない（`/qa` と同じ降格ルール）
- テストコードは書かない（恒常的な回帰テスト化が必要なら `/test` を案内する）

## 完了条件

レポート `tests/qa/recon/<date>-<機能名>.md` の記録と、（台帳があれば）ケース状態の反映が終わり、一致/不一致サマリと要確認件数を報告した時点で完了。不一致があれば人間判断（台帳の `要確認` 解消）を次アクションとして案内する。
