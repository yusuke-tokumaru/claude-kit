# implement

確定した計画、または方向性が明確なタスクを**規律をもって実装する**スキル。テストは後置きを既定とし、実装の駆動に集中する。

**位置づけ**: `/plan` が計画を確定したあとの実行フェーズを担う（plan から委譲される）。計画なしの明確な小タスクなら単独でも起動できる。実装完了後は `/test` → `/review` → `/pr` の既存チェーンへ。

**コア原則: 各ステップは緑（検証通過）になるまで次へ進まない。動いていない状態を積み上げない。**

## 前提

- **計画は作らない**。設計判断や手順の確定が必要なら `/plan` の責務。implement は確定済みの方針を実装に落とす。
- **既定では新規テストを書かない**。テストの作成は `/test` の責務（テスト後置き）。implement の検証は既存テスト・型チェック・実挙動で行う（例外は Step 2 の TDD モード）。

**非対話実行（サブエージェント・自動実行などユーザーに質問できない場合）**: 「ユーザーに確認」「合意したら」のステップは推奨案を採用して進めてよい。ただし置いた仮定は最終報告で「本来ユーザーに確認すべきだった点」として列挙する。3回失敗ルール等の規律は非対話でも緩和されない。

## Step 1: 実装単位を確定する

- **plan 経由の場合**: `plans/<date>-<slug>.md`（または承認済みプラン）のステップ列をそのまま実装単位にする。共有コンテキスト・依存チェーン・各ステップの検証基準を読み込む。
- **単独起動の場合**: `$ARGUMENTS` のタスクを、コミット可能な最小ステップに分解する。各ステップに「何を変えるか」「どう検証するか」を割り当てる。分解が自明でない＝設計判断が要るなら、その場で `/plan` を案内して止まる（implement は計画しない）。
- 着手前に `git status --short` で作業ツリーの状態を確認する。

## Step 2: 進め方を確認する（テスト姿勢・コミット方針）

着手前に、次の2点を1度だけ確認する。

**テスト姿勢**（既定=テスト後置き）: テストの本作成は後段 `/test` に委ね、implement では既存テスト＋型/lint＋実挙動で各ステップを検証する。ユーザーが「TDD で」と明示した場合のみ**テスト先行モード**に切り替える（各ステップで失敗するテストを先に書く → 最小実装で通す → リファクタ。この場合のみ implement がテストを書く）。

**コミット方針**（`AskUserQuestion` で1度だけ確認する）:
- **各ステップでコミットする**: 検証が緑になったステップごとに命令形メッセージでコミットする（無関係ファイルはステージしない）。
- **コミットしない（既定）**: 実装・検証・マーカー更新のみ行い、コミットは後でユーザーまたは `/pr` に委ねる。

非対話時はテスト後置き＋「コミットしない」を既定にする（キットの「`git add` を自動実行しない」規約に従う）。

## Step 3: ステップを1つずつ実装する

各ステップで以下を厳守する（**1ステップ = 1コミット = 1検証**）。

1. **最小差分**で実装する。ステップの目的を満たす変更だけ。「ついでにリファクタ」「ついでに別の改善」をバンドルしない。
2. **既存パターンを踏襲する**。周辺コードの命名・構造・イディオム・エラーハンドリングに合わせる（plan/kickoff が調べた Patterns があれば入力にする）。新規の流儀を持ち込まない。
3. **検証する**（テストを書かずに動作を確認する）:
   - 既存テストスイートを実行してリグレッションが無いか
   - 型チェック・lint（プロジェクトにあれば）
   - 実際に動かす（CLI 実行・画面操作・関数呼び出し）。UI なら必要に応じて `playwright-mcp-e2e` の手順で視覚確認
4. 検証が緑になったら、plan 経由ならプランの進捗マーカーを `- [ ]` → `- [x]` に更新する。Step 2 のコミット方針が「各ステップでコミットする」ならここでコミットする（命令形メッセージ・無関係ファイル除外）。
5. 次のステップへ進む。

## Red Flags — STOP

これらの思考・状況になったら止まる:

- 「まとめて変えて最後に確認」→ ステップ単位で検証する。どの変更が壊したか分からなくなる
- 「テストが邪魔だから実装に合わせて書き換える」→ テストに合わせて実装を歪めない（正しくロジックを直すのは可）
- 「ついでにこれも直す」→ バンドル禁止。別ステップ・別コミットにする
- **同一ステップの修正が3回失敗** → 4回目を試さない。計画（そのステップの設計）を疑い、ユーザーと議論する。plan 経由なら `/plan` に差し戻す
- 「これは設計判断だが勝手に決めて進む」→ 方針が未確定なら止まって `/plan` を案内する

## 制約

- 新規テストを書かない（テスト後置きモード時）。テスト作成は `/test` の責務
- 計画・設計判断をこのスキルで確定しない。`/plan` の責務
- テスト通過のためだけに実装コードを歪めない
- 既存テストを削除・改変して「通す」ことをしない
- コミット方針は着手時に1度だけ確認する（既定=コミットしない＝`git add` を自動実行しない）。push はユーザー承認後のみ。コミットする場合も無関係ファイルをステージしない

## 完了条件

全ステップが実装・検証済み（plan 経由なら全マーカーが `- [x]`）で、既存テスト・型チェックが通過している状態。完了後は `/test`（後置きテストの本作成）→ `/review` を案内する。
