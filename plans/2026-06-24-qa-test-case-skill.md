# ck:qa スキル実装プラン

## タスク要約

「動いているはず、を信用せず意地悪に疑う」スタンスで QA テストケース台帳を生成する新スキル `ck:qa` を、二層構造（本体 `packages/ck/src/skills/qa/body.md` ＋ スタブ `plugins/ck/skills/qa/SKILL.md`）で追加する。正常系バイアスを P1〜P7 ペルソナの各最低1観点で潰し、新機能/移行の二軸でオラクルを切り替え、機能単位 Markdown 台帳＋2階層サマリで陳腐化・回帰属人化を防ぐ。

- **ブランチ**: 現在のブランチ（main）で続行
- **成果物**: スキル2ファイル ＋ CLAUDE.md のストレージ規約追記。`tests/qa/` はスキル実行時に生成される（本プランでは生成しない）。

## 調査サマリー

- **スキル追加は登録不要**: `ck skill list/print` は `SKILLS_DIR = packages/ck/src/skills` をディレクトリスキャンするだけ。`cli.ts` やレジストリへの追記は不要。`packages/ck/src/skills/qa/body.md` と `plugins/ck/skills/qa/SKILL.md` を置けば動く。
- **新スタブ認識にはセッション再起動が必要**（既存スキルの本体変更は即反映）。
- **メタデータの源泉はスタブ側のみ**。body.md にフロントマターを書くと `ck skill doctor` がエラーにする（`print` は body 先頭の `---...---` を除去して出力）。
- **doctor の検査**: スタブ⇔body ディレクトリ対応 / `name:` とディレクトリ名一致 / `description:` 存在 / 実行行 `!ck skill print <name>` の name 一致 / フォールバック `|| echo` がバッククォート内にあるか。
- **同型の参考スキル**: `ck:review`・`ck:debug`（CLI 委譲系、Step/Phase 構成・制約・完了条件を持つ）。

## ステップ間共有コンテキスト

**スキル名**: `qa`（ディレクトリ名・`name:`・`ck skill print qa` すべて一致必須）

**ペルソナ固定リスト（P1〜P7）**:
```
P1 新人ユーザー        : 説明を読まず直感操作。誤クリック・空送信・連打で壊れないか
P2 ベテラン現場担当    : 高速・大量入力。Tab遷移/ショートカット/IME変換中Enter
P3 悪意ある操作者      : 境界値・不正値・権限外・二重送信。バリデーション/排他制御
P4 データ整合性監査役  : 画面を信用せず裏のDBテーブルを直接確認。CRUD整合性
P5 移行担当者          : 旧データ投入。欠損・異形式・文字コード・件数一致
P6 回帰デグレ番人      : 既存機能が壊れていないか。周辺機能・F5後の挙動
P7 仕様懐疑者          : 実装＝正しい仕様を信用しない。一次情報と挙動の突合
```

**状態3値**: `active`（有効）/ `要確認`（仕様変更で要再判定・移行の旧バグ疑い）/ `obsolete`（廃止、削除せず残す）

**台帳ファイルパス**: `tests/qa/<機能名>.md`（kebab-case）、横断 index は `tests/qa/README.md`

**台帳フォーマット（body.md に埋め込むテンプレート）**:
```markdown
# <機能名> QAテストケース台帳
- モード: 新機能 | 移行
- 一次情報: <課題#・ソースパス / 旧仕様書・旧挙動>
- ケース総数: N（active A / 要確認 W / obsolete O）
- ペルソナ別: P1×n P2×n P3×n P4×n P5×n P6×n P7×n
- 要確認: W件 / 最終更新: YYYY-MM-DD

## テストケース
| ID | ペルソナ | 観点 | 前提/手順 | 期待結果（裏側確認含む） | 関連ファイル | 状態 |
|----|---------|------|----------|------------------------|------------|------|
| C1 | P3 | 数量0送信 | 数量欄に0で送信 | バリデーションで拒否、orders未作成 | src/order/validate.ts | active |

## 回帰トリガー（P6）
- <この機能の◯◯を変更したら → 確認すべきケースID / 他台帳のケース>

## 該当なしペルソナと理由
- P5: 本機能は移行対象外のため該当なし
```

**横断 index フォーマット（`tests/qa/README.md`）**:
```markdown
# QAテストケース台帳 index
| 機能 | 台帳 | active | 要確認 | obsolete | 更新日 |
|------|------|--------|--------|----------|--------|
| 受注入力 | [order-entry.md](order-entry.md) | 10 | 1 | 2 | 2026-06-24 |
```

## ステップ

### - [x] Step 1: スタブ SKILL.md を作成

**コミット**: `feat: ck:qa スキルのスタブを追加`

**ファイル**: `plugins/ck/skills/qa/SKILL.md`（新規）
```yaml
---
name: qa
description: テストケース・QA観点の洗い出しが必要なときに使用する。「テストケースを作って」「QA観点を出して」「移行データの確認項目を洗い出して」と言われたとき。正常系に偏らずP1〜P7ペルソナで網羅する。
user-invocable: true
allowed-tools: Bash, Read, Grep, Glob, Task, AskUserQuestion, Edit, Write
argument-hint: <機能名 or 対象>
---

!`ck skill print qa || echo "ERROR: ck not found. Fix: cd ~/claude-kit && bun link"`
```

**依存**: なし（最初のステップ）
**生産物**: スタブ。これにより `ck skill doctor` の対応検査対象になる。
**検証**: `ck skill doctor` がこのスタブについて「body.md なし」以外のフロントマター系エラーを出さないこと（body は Step 2 で作る）。

> **Why（description のトリガー条件化）**: 直近コミット `165a349 fix: スキルの…description のトリガー条件化` の方針に合わせ、「いつ使うか」を具体的な発話例で書く。
> **Why（allowed-tools）**: モード選択に AskUserQuestion、一次情報収集に Task(Explore)/Read/Grep/Glob、変更ファイル逆引きや既存台帳確認に Bash、台帳生成に Write/Edit が必要。debug と同じフルセット。

### - [x] Step 2: 本体 body.md を作成

**コミット**: `feat: ck:qa スキル本体（P1〜P7ペルソナ・二軸モード・台帳生成）を追加`

**ファイル**: `packages/ck/src/skills/qa/body.md`（新規、フロントマターなし）

**章構成**（review/debug と同型の CLI 委譲系）:
```
# qa
（位置づけ段落: 「何をテストすべきか」=テストケース台帳を生む。コードは生まない。
  /test（ユニットコード）・/playwright-mcp-e2e（E2Eコード）の前段または並走。
  共通スタンス: 動いているはずを信用せず意地悪に疑う。）

## Step 1: モード選択          ← AskUserQuestion で 新機能 / 移行 を明示選択
## Step 2: 一次情報の収集        ← モード別オラクル。Task(Explore)/Read/Grep で課題・ソース or 旧仕様・旧挙動・旧データ
## Step 3: 既存台帳の照合        ← tests/qa/<機能名>.md があれば読み、現在の一次情報と差分照合し 要確認/obsolete に降格
## Step 4: ペルソナ走査          ← P1〜P7 各最低1観点。無ければ「該当なし＋理由」。P4は裏側確認を必須
## Step 5: 台帳の生成・更新      ← tests/qa/<機能名>.md を Write/Edit。回帰トリガー節・関連ファイルタグ込み
## Step 6: サマリ更新            ← ファイル先頭ヘッダ（ペルソナ別カバレッジ含む）＋ tests/qa/README.md index 更新

## 台帳フォーマット             ← 共有コンテキストのテンプレートを掲載
## ペルソナ定義（P1〜P7）        ← 共有コンテキストのリスト。各ペルソナの「疑う点」と最低1観点ルール
## 状態定義（active/要確認/obsolete）
## モード別オラクル早見表        ← 新機能=あるべき仕様 / 移行=旧挙動一致＋旧バグは要確認
## 制約
## 完了条件
```

**「制約」節に必ず含める**:
- 実在の顧客名・案件名・認証情報を台帳に書かない（CLAUDE.md 運用ルール）
- 既存ケースを勝手に削除しない（要確認/obsolete に降格し人間判断へ）
- テストコードは生成しない（/test・/playwright-mcp-e2e へ委譲）
- 各ペルソナの観点漏れを黙殺しない（該当なしは理由明記）

**「完了条件」節**: 台帳ファイルとサマリ（先頭ヘッダ＋index）を更新し、P1〜P7 全てが観点 or 該当なし理由を持つ状態。次に必要なら /test・/playwright-mcp-e2e を案内。

**依存**: Step 1（スタブ）。`ck skill print qa` が本体を読めるようになる。
**生産物**: スキル本体。
**検証**: `ck skill print qa` が本体を出力（フロントマター混入なし）。`ck skill doctor` がエラー0。

## 議論が必要な設計決定

### 設計決定 A: スキル名 — `qa` で確定するか
- 候補: `qa`（簡潔・推奨）/ `qa-cases` / `test-cases`
- トレードオフ: `qa` は短く `/qa` で呼べるが範囲がやや広い印象。`qa-cases` は意図が明確だが冗長。
- **推奨**: `qa`。decisions でも `ck:qa（仮）` 表記。

### 設計決定 B: allowed-tools のスコープ
- 案1（推奨）: `Bash` 無制限（debug と同様フルセット）。一次情報収集や変更ファイル逆引きで柔軟。
- 案2: `Bash(git *, ck *)` などサブコマンド限定。最小権限だが、スキーマ確認等で制限に当たる懸念。
- **推奨**: 案1。本スキルは調査比重が高く debug と同類。

## リスクと軽減策

- **リスク1（観点の形骸化）**: P1〜P7 を機械的に埋めるだけで中身が正常系に寄る恐れ。
  - **軽減策**: body.md の各ペルソナ定義に「疑う点」を具体例付きで固定し、P4 は期待結果に裏側テーブル確認を**必須**と明記。サマリのペルソナ別カバレッジで偏りを数値可視化。
- **リスク2（新スタブ未認識）**: セッション中に作っても Claude Code がスタブを認識しない。
  - **軽減策**: Step 4 で再起動が必要な旨を案内。本体（body.md）変更は即反映なので開発反復は可能。
- **リスク3（照合パスの誤降格）**: 再実行時に有効なケースを誤って obsolete にする。
  - **軽減策**: 状態は3値で、削除せず `要確認`/`obsolete` 降格に留め最終判断を人間に残す（制約節に明記）。

## ロールバック

スキル追加のみで既存挙動に影響しないため、`plugins/ck/skills/qa/` と `packages/ck/src/skills/qa/` を削除すれば原状復帰。CLAUDE.md は該当追記を revert。
