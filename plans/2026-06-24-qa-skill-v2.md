## ステップ間共有コンテキスト

**台帳テーブル列（9列・確定）**
```
| ID | レンズ | 技法 | 優先度 | 観点 | 前提/手順 | 期待結果（裏側確認含む） | 関連ファイル | 状態 |
```

**検証レンズ L1〜L7（旧ペルソナ対応）**
```
L1 直感操作で壊す（旧P1新人）        L5 既存データ投入で壊す（旧P5移行担当）
L2 高速大量入力で壊す（旧P2ベテラン）  L6 周辺の回帰で壊す（旧P6回帰番人）
L3 境界・不正・権限で壊す（旧P3）      L7 仕様との不一致を暴く（旧P7仕様懐疑者）
L4 裏のDB整合を疑う（旧P4監査役）
```
- 重点配分: 新機能=L1〜L4 厚め / 移行=L5〜L7 厚め。ただし全レンズ一巡。
- L4 は期待結果に裏側DB確認を必須化（旧P4の規約を継承）。
- 回帰トリガー節は L6 担当（旧P6）。

**採用5技法（固定）**
```
境界値分析 / 同値分割 / 状態遷移 / デシジョンテーブル / 組合せ・ペアワイズ
```

**優先度（3値）**
```
High = データ破壊・金額・権限に関わる / Mid = 主要導線 / Low = 表示・軽微
```

**ケースID書式**: `<機能略号>-C<連番>`（例 `ORD-C1`、`STK-C3`）。略号は kebab-case 機能名から導出し、台帳ヘッダで宣言。対話可能ならユーザー確認、非対話なら AI が機能名から推定し仮定としてヘッダに明記。

**台帳ヘッダ（更新後イメージ）**
```
- モード: 新機能 | 移行
- 機能略号: ORD
- 一次情報: ...
- ケース総数: N（active A / 要確認 W / obsolete O）
- レンズ別: L1×n L2×n L3×n L4×n L5×n L6×n L7×n
- 技法別: 境界値×n 同値×n 状態遷移×n デシジョン×n 組合せ×n
- 要確認: W件 / 最終更新: YYYY-MM-DD
```
